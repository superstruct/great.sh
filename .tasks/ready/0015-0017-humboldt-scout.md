## Risks

### Risk 1: `/osxsdk` separate directory (ADVISORY, Socrates concern #1)
The upstream osxcross image places files at both `/osxcross` and potentially `/osxsdk`. The spec only `COPY --from=osxcross /osxcross /osxcross`. If SDK headers live at `/osxsdk` separately, the build will compile but fail at link time with missing SDK symbols. The upstream README only references `/osxcross`, so this is likely not an issue — but the builder should inspect the image layout after task 0015 build if any link errors appear.

Mitigation: if `cargo build` fails with missing headers after the Dockerfile change, add `COPY --from=osxcross /osxsdk /osxsdk` as a second line after the first COPY.

### Risk 2: osxcross `PATH` may shadow system `clang` (ADVISORY, Socrates concern #5)
The new `ENV PATH="/osxcross/bin:${PATH}"` is added before apt-get installs `clang` and `lld`. osxcross binaries are prefixed with target triples (e.g., `x86_64-apple-darwin26.1-clang`), so bare `clang` is unlikely to be shadowed. Low risk.

### Risk 3: Image size
The resulting macOS cross image will be approximately 2-4 GB (osxcross toolchain + SDK + Ubuntu base + Rust). This is expected for a local-only development tool. It is never pushed to a registry in current workflows.

### Risk 4: `cross-linux-aarch64.Dockerfile` has same broken Rust version (tech debt)
`docker/cross-linux-aarch64.Dockerfile` line 6 = `FROM rust:1.83-slim`. Not in scope now, but will break on first use. File a follow-on task after 0015+0017 are done.

### Risk 5: `test-files/` directory write permissions
`test-files/` is currently empty and is mounted as read-write from docker-compose. The directory exists on the host. No risk — the test scripts will `mkdir -p /workspace/test-files` before writing, which is redundant but safe.
