---

## 1. Problem Statement

When `great` runs inside a Docker container hosted on a WSL2 machine, the platform detection functions `is_wsl()` and `is_wsl2()` in `/home/isaac/src/sh.great/src/platform/detection.rs` return `true` because Docker containers on WSL2 inherit the host kernel's `/proc/sys/fs/binfmt_misc/WSLInterop` file and `/proc/version` string (which contains "microsoft"). This causes the CLI to classify the environment as `Platform::Wsl` rather than `Platform::Linux`, leading `great apply` to attempt WSL-specific actions (copying fonts to Windows paths, invoking `cmd.exe`) that fail inside the container. The fix is to add a container-detection guard that short-circuits the WSL checks when the environment is detected as a container.

---

## 2. Solution Design

### 2.1 Strategy

Add a new private helper function `is_container()` (and its testable counterpart `is_container_with_probe()`) that checks four container indicators in priority order. Insert a call to this helper at the top of both `is_wsl()` / `is_wsl_with_probe()` and `is_wsl2()` / `is_wsl2_with_probe()`. If any container indicator is present, those functions return `false` immediately, bypassing all WSL probes.

### 2.2 Container Indicators (checked in this order)

| Priority | Indicator | Probe Method | Rationale |
|----------|-----------|--------------|-----------|
| 1 | `/.dockerenv` file exists | `path_exists("/.dockerenv")` | Docker creates this sentinel file in every container. Most reliable single indicator. |
| 2 | `DOCKER_CONTAINER` env var is set | `env_var("DOCKER_CONTAINER")` | Set by some Docker Compose configurations and base images. |
| 3 | `container` env var is set | `env_var("container")` | Set by Podman, systemd-nspawn, and other OCI runtimes. Lowercase `container` is the standard name. |
| 4 | `/proc/1/cgroup` contains "docker" | `read_file("/proc/1/cgroup")` then `contains("docker")` | Cgroup v1 fallback. On cgroup v2, this file may not contain "docker", so it is the last resort. |

The check is short-circuit: as soon as any indicator matches, the function returns `true` without evaluating subsequent indicators.

### 2.3 Why No `OsProbe` Trait Extension Is Needed

The existing `OsProbe` trait (line 295) already provides the three methods needed:
- `path_exists(&self, path: &str) -> bool` -- for `/.dockerenv`
- `env_var(&self, name: &str) -> Option<String>` -- for `DOCKER_CONTAINER` and `container`
- `read_file(&self, path: &str) -> Option<String>` -- for `/proc/1/cgroup`

No new trait methods are required.

## 3. Exact Changes

All changes are in a single file: `/home/isaac/src/sh.great/src/platform/detection.rs`.

### 3.1 New Production Function: `is_container()`

Insert after line 189 (after the closing brace of `fn is_wsl2()`), before `fn detect_linux_distro()`:

```rust
/// Returns `true` when running inside a container (Docker, Podman, etc.).
///
/// Checks four indicators in priority order: /.dockerenv file, DOCKER_CONTAINER
/// env var, container env var (OCI standard), and /proc/1/cgroup contents.
fn is_container() -> bool {
    if std::path::Path::new("/.dockerenv").exists() {
        return true;
    }

    if std::env::var("DOCKER_CONTAINER").is_ok() {
        return true;
    }

    if std::env::var("container").is_ok() {
        return true;
    }

    std::fs::read_to_string("/proc/1/cgroup")
        .map(|c| c.contains("docker"))
        .unwrap_or(false)
}
```

## 6. Edge Cases

| Edge Case | Behavior | Covered By |
|-----------|----------|------------|
| `/.dockerenv` exists but is empty (0 bytes) | `path_exists` returns `true` regardless of file content. Container detected. | Test 25 (paths entry is boolean; content irrelevant) |
| `container` env var is set to empty string `""` | `env_var()` returns `Some("")`. Container detected. This is correct: the env var's mere presence signals a container. | Test 27 (value does not matter, only presence) |
| `/proc/1/cgroup` is unreadable (permission denied) | `read_file` returns `None`. Falls through to `unwrap_or(false)`. Container NOT detected via this indicator, but earlier indicators may still trigger. | Test 29 (no files set, returns false) |
| `/proc/1/cgroup` exists but is empty | `contains("docker")` returns `false`. Container not detected via cgroup. | Implicit: empty string does not contain "docker" |
| Cgroup v2 system: `/proc/1/cgroup` contains `0::/` only | `contains("docker")` returns `false`. Container not detected via cgroup alone. Other indicators (`/.dockerenv`) still work. | Test 30 (uses `/.dockerenv` instead) |
| WSL1 (no WSLInterop, `/proc/version` has "Microsoft") inside Docker | Container guard triggers first, returns `false` from `is_wsl`. Correct behavior. | Test 31 (all tiers plus container indicator) |
| Multiple container indicators present simultaneously | Short-circuit evaluation: first matching indicator wins. No double-counting. | All container tests check a single indicator; Test 31 has container + WSL |
| Genuine WSL2 with Docker installed (but not inside a container) | `/.dockerenv` does not exist on the host. `DOCKER_CONTAINER` and `container` env vars are not set. `/proc/1/cgroup` does not contain "docker" (host PID 1 is init/systemd). `is_container()` returns `false`. WSL detection proceeds normally. | Test 33 |
| LXC/systemd-nspawn container on WSL2 | The `container` env var is set by both LXC and systemd-nspawn. Container guard triggers. | Test 27 |
| Podman rootless container on WSL2 | Podman sets the `container` env var. Container guard triggers. | Test 27 |

## 7. Error Handling

This change involves no fallible operations that produce user-visible errors. All checks are boolean predicates:
- `path_exists` returns `false` on any error (permission denied, path not found).
- `env_var` returns `None` if the variable is absent or not valid Unicode.
- `read_file` returns `None` on any I/O error; the `unwrap_or(false)` fallback ensures graceful degradation.

If all four container indicators fail to trigger (e.g., a novel container runtime that sets none of them), the behavior degrades to the current (pre-fix) behavior: the environment will be classified as WSL. This is an acceptable degradation -- it is no worse than the status quo and can be improved in a future task by adding more indicators.

No new error types, error messages, or `anyhow::Result` returns are introduced.

## 8. Security Considerations

- **Indicator spoofing:** A non-container environment could create `/.dockerenv` or set `DOCKER_CONTAINER`/`container` env vars, causing the container guard to suppress WSL detection. This would result in classifying a genuine WSL environment as plain Linux -- a safe failure mode (Linux behavior is a subset of WSL behavior, and no WSL-specific actions would be attempted).
- **Container escape detection is out of scope:** This fix does not attempt to detect whether a container has escaped its sandbox. It only checks positive container indicators.
- **No new filesystem writes or network access:** All operations are read-only.
- **No new dependencies:** The fix uses only `std::path::Path::exists()`, `std::env::var()`, and `std::fs::read_to_string()` -- all already used by the existing detection code.

## 10. Out of Scope

- Detecting container runtimes beyond Docker/Podman/OCI (LXC and systemd-nspawn are covered via the `container` env var, but exotic runtimes like Kata or gVisor are not targeted).
- Adding a `Platform::Container` variant to the `Platform` enum. The container is classified as `Platform::Linux`, which is correct.
- Changes to downstream consumers (`apply.rs`, `doctor.rs`, `status.rs`). Fixing the detection function fixes all consumers automatically.
- Adding `is_container()` to the public API. The function is private (`fn`, not `pub fn`). If other modules need container detection in the future, it can be promoted in a separate task.
- Cgroup v2 parsing. On pure cgroup v2 systems, `/proc/1/cgroup` typically contains only `0::/` without "docker". The `/.dockerenv` file and env var checks cover this case. A dedicated cgroup v2 detection heuristic (e.g., checking `/proc/self/mountinfo` for overlay mounts) is deferred.
