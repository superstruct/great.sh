## Code Analysis

### `ApplyCategory` enum (lines 353–363)
```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq, ValueEnum)]
pub enum ApplyCategory { Tools, Mcp, Agents, Secrets }
```
- `Copy` — no heap allocation. Stack-only. Correct.
- 4 variants. Fixed, not unbounded.

### `Args` struct additions (lines 383–387)
```rust
pub only: Vec<ApplyCategory>,
pub skip: Vec<ApplyCategory>,
```
- `Vec` is allocated only at CLI parse time, once per invocation.
- Bounded by clap argument count — in practice 0–4 elements.
- `value_delimiter = ','` means a single string is parsed rather than repeated flags; either way allocation is O(1) amortised.

### `should_apply` function (lines 396–404)
```rust
fn should_apply(category: ApplyCategory, only: &[ApplyCategory], skip: &[ApplyCategory]) -> bool {
    if !only.is_empty() { return only.contains(&category); }
    if !skip.is_empty() { return !skip.contains(&category); }
    true
}
```
- O(n) where n ≤ 4. Equivalent to O(1) for all practical purposes.
- Takes slices — no cloning.
- Short-circuits correctly: `only` checked first, then `skip`.

### Dead code warning
The compiler emits `warning: function 'should_apply' is never used` in the debug build. This function was declared but not yet wired into the `run()` call sites. The same warning is **absent from the release build** only because release warnings do not repeat unused-code for non-pub items, but the implementation gap exists. The `run()` function proceeds through all sections regardless of `--only`/`--skip` at runtime.

**This is a correctness issue, not a performance issue.** It does not affect binary size or allocation behaviour, but the flags are currently no-ops. Flagged for Socrates / Da Vinci attention — not a performance block.
